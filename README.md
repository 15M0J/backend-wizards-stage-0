# api-practice-lab

Stage 2 Backend Wizards profile API for Insighta Labs. The service stores demographic profiles in PostgreSQL and supports filtering, sorting, pagination, and rule-based natural language search.

## Stack

- Node.js 22
- Express
- PostgreSQL via `pg`
- Axios for the optional Stage 1 enrichment endpoint
- Vercel-compatible serverless entrypoint

## Environment

Required:

- `DATABASE_URL`

Optional:

- `POSTGRES_URL`
- `PORT`

## Run locally

```bash
npm install
npm start
```

The local server runs on `http://localhost:3000` by default.

## Seed profiles

Seed the provided 2026 profile JSON file before submitting:

```bash
npm run seed -- ./profiles-2026.json
```

The seed command also accepts an HTTP(S) URL:

```bash
npm run seed -- https://example.com/profiles-2026.json
```

Seeding is idempotent. Profiles are keyed by unique normalized `name`, so re-running the seed updates existing rows instead of creating duplicates.

## Database model

The app creates and migrates the `profiles` table on first request:

- `id` UUID v7 primary key
- `name` VARCHAR unique
- `gender` VARCHAR
- `gender_probability` FLOAT
- `age` INT
- `age_group` VARCHAR
- `country_id` VARCHAR(2)
- `country_name` VARCHAR
- `country_probability` FLOAT
- `created_at` TIMESTAMP generated by the database

Indexes are created for the main filter and sort fields: `gender`, `age_group`, `country_id`, `age`, `gender_probability`, `country_probability`, and `created_at`.

## API

All responses include `Access-Control-Allow-Origin: *`.

### `GET /api/profiles`

Returns filtered, sorted, paginated profiles.

Supported filters:

- `gender`: `male` or `female`
- `age_group`: `child`, `teenager`, `adult`, `senior`
- `country_id`: two-letter ISO country code
- `min_age`
- `max_age`
- `min_gender_probability`
- `min_country_probability`

Sorting:

- `sort_by`: `age`, `created_at`, or `gender_probability`
- `order`: `asc` or `desc`

Pagination:

- `page`: default `1`
- `limit`: default `10`, max `50`

Example:

```txt
/api/profiles?gender=male&country_id=NG&min_age=25&sort_by=age&order=desc&page=1&limit=10
```

Response:

```json
{
  "status": "success",
  "page": 1,
  "limit": 10,
  "total": 2026,
  "data": []
}
```

### `GET /api/profiles/search`

Runs a rule-based natural language search. Pagination with `page` and `limit` is supported.

Example:

```txt
/api/profiles/search?q=young males from nigeria&page=1&limit=10
```

## Natural language parsing approach

The parser is intentionally rule-based and does not use AI or LLM calls. It lowercases the query, strips punctuation, tokenizes words, then builds the same filter object used by `GET /api/profiles`.

Supported keyword mappings:

- Gender: `male`, `males`, `man`, `men`, `boy`, `boys` map to `gender=male`.
- Gender: `female`, `females`, `woman`, `women`, `girl`, `girls` map to `gender=female`.
- If both male and female terms appear, no gender filter is applied. Example: `male and female teenagers above 17` maps only to `age_group=teenager&min_age=17`.
- Age group: `child`, `children`, `kids` map to `age_group=child`.
- Age group: `teenager`, `teenagers`, `teen`, `teens` map to `age_group=teenager`.
- Age group: `adult`, `adults` map to `age_group=adult`.
- Age group: `senior`, `seniors`, `elderly` map to `age_group=senior`.
- `young` maps to `min_age=16&max_age=24` for search only. It is not stored as an `age_group`.
- Age ranges: `between 18 and 30` maps to `min_age=18&max_age=30`.
- Lower bounds: `above 30`, `over 30`, `older than 30`, `at least 30`, `minimum age 30`, and `min age 30` map to `min_age=30`.
- Upper bounds: `below 30`, `under 30`, `younger than 30`, `less than 30`, `at most 30`, `maximum age 30`, and `max age 30` map to `max_age=30`.
- Exact age: `age 25` or `aged 25` maps to `min_age=25&max_age=25`.
- Countries: country names, common aliases, and two-letter codes after `from` or `in` map to ISO country codes. Examples: `nigeria` to `NG`, `angola` to `AO`, `kenya` to `KE`, `from ng` to `NG`, `usa` to `US`, and `uk` to `GB`.

If no supported keyword, age expression, or country can be found, the endpoint returns:

```json
{ "status": "error", "message": "Unable to interpret query" }
```

## Parser limitations

- The parser does not understand grammar beyond fixed keywords and regular expressions.
- It does not support negation such as `not male` or `not from Nigeria`.
- It does not infer related concepts such as `west african`, `college age`, or `working age`.
- It does not support compound alternatives except the explicit male-and-female behavior. For example, `from Nigeria or Kenya` will match the first recognized country only.
- Country matching is alias-based, so obscure aliases and demonyms such as `Nigerian` or `Kenyan` are not guaranteed.
- Age comparisons are inclusive because the grading examples map `above 30` to `min_age=30`.

## Other endpoints

### `POST /api/profiles`

Creates one profile by name using Genderize, Agify, and Nationalize enrichment.

```json
{ "name": "ella" }
```

### `GET /api/profiles/:id`

Returns one profile by UUID.

### `DELETE /api/profiles/:id`

Deletes one profile by UUID and returns `204 No Content`.

## Error responses

All API errors use this shape:

```json
{
  "status": "error",
  "message": "..."
}
```

Common errors:

- `400 Bad Request`: missing or empty required query/body parameter, unrecognized query parameters, or uninterpretable natural language query
- `422 Unprocessable Entity`: invalid parameter type, invalid enum value, invalid number, or invalid range
- `404 Not Found`: profile not found
- `502 Bad Gateway`: upstream enrichment API failure
- `500 Internal Server Error`: unexpected server or database failure

Invalid query parameters return:

```json
{ "status": "error", "message": "Invalid query parameters" }
```

## Deployment

1. Create a hosted PostgreSQL database.
2. Add `DATABASE_URL` or `POSTGRES_URL` in the deployment environment.
3. Deploy to Vercel or another supported platform.
4. Run the seed command against the production database.
5. Submit the API base URL and GitHub repo URL.
